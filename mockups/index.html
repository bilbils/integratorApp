AI Agents

Each agent is a saved job: a name, a prompt, a model, and who can call it. Add a new capability by adding a row — no deploy. Click an agent to edit it.

Aggregators

Inbound data connectors — Plaid, Nango, Plaud. Different from AI Agents: these hold credentials and a data-handling rule, not a prompt. Same access-grant spine underneath. Click one to edit.

Consumer Apps

The registered LFODIE apps that can call agents / read aggregators — each with its own key.

Consumer Apps — part of this same admin

Staffility · Bills-Master-Plan · IPTA tools · Blue Orbit. Register an app, issue or rotate its key, and see what it's allowed to touch. Sketched out next to the other screens.

Access Grants

The grid that ties it all together: which app may call which agent and read which aggregator.

Access Grants — the connective tissue

One simple matrix: apps down the side, agents + aggregators across the top, a checkbox where they meet. This is where "available to all, private to each" actually gets enforced.

How to read this: This is a clickable mockup, not the real app — edits live only in this page. It shows the shape of the AI-Agents registry we agreed on: cheapest-capable model by default, an optional stronger fallback per agent, per-app access, and a cost/outcome panel so you escalate a model on evidence, not by guessing. Model names shown are illustrative — the real list comes from whatever gateway sits underneath (LiteLLM / OpenRouter), and a model can be hosted today or Staffility-hosted later by changing one field.
